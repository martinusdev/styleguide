§ 01 · Koncept

Čo je Model Context Protocol?

Veľké jazykové modely vedia rozprávať, ale samé osebe sú v izolácii. Nevidia naše súbory, databázy, ani styleguide. MCP je protokol, ktorý túto bariéru ruší — definuje, ako môže LLM volať externé nástroje a získavať dáta z reálneho prostredia.

Pred MCP každý vendor (OpenAI, Google, Anthropic) riešil prepojenie modelu s nástrojmi po svojom — proprietárne function calling API. Výsledok? Roztrieštený ekosystém, integrácie sa nedali zdieľať medzi modelmi, každá appka si musela písať vlastné lepidlo.

V novembri 2024 Anthropic publikoval MCP ako open-source, vendor-neutrálny štandard. Princíp je rovnaký ako pri Language Server Protocole (LSP) pre IDE — jeden protokol, ktorý hovoria všetci.

Inšpiráciou pre MCP bol LSP — ten istý prístup, ktorý umožnil, že rovnaký jazykový server beží vo VS Code, Vim, IntelliJ aj Sublime.

§ 02 · Primitíva

MCP server ponúka tri veci.

Tools, resources, prompts — tri stavebné bloky každého servera. My zatiaľ používame len tools.

U nás 01 · Aktívne

Tools

Funkcie, ktoré LLM volá s parametrami. Server ich vykoná a vráti výsledok. Najpoužívanejšie primitívum.

get_component(name: "buttons")

Pasívne 02 · Dáta

Resources

Dáta, ktoré server vystavuje na čítanie — súbory, log záznamy. LLM ich číta podľa URI.

file:///app/main.scss

Šablóny 03 · Prompty

Prompts

Pripravené prompt šablóny s parametrami, ktoré klient ponúka ako rýchle akcie.

"Vytvor formulár pre {{entita}}"

Transport. MCP definuje dva transporty: lokálny stdio (server beží ako proces, komunikuje cez pipy) a vzdialený HTTP so server-sent events. Aplikačná vrstva je JSON-RPC 2.0.
§ 03 · Lifecycle

Anatómia jedného volania.

Čo sa stane medzi promptom a odpoveďou. Discovery (1–2) sa deje raz pri pripojení; tool call (3–5) môže prebehnúť mnohokrát.

1 initialize

Handshake medzi klientom a serverom — výmena verzií a capabilities.

2 tools/list

Klient si vypýta zoznam dostupných toolov so schemami parametrov.

3 LLM rozhoduje

Claude na základe promptu vyberá, ktorý tool zavolá a s akými argumentmi.

4 tools/call

Klient pošle request s názvom toolu a parametrami. Server validuje a vykoná.

5 response

Server vráti štruktúrovaný výsledok, ktorý LLM použije v odpovedi.

Tip. Celý cyklus prebehne v jednej konverzácii niekoľkokrát. Server beží stále — žiadny cold start.

§ 04 · Praktiky

Čo funguje, čo nie.

MCP je jednoduchý protokol, ale postaviť dobrý server je iný príbeh. LLM nečíta kód — číta popisy toolov.

 Funguje

  • Stručné, popisné názvy — get_component, list_icons. LLM ich číta ako API.
  • Striktné JSON schema pre parametre s enum hodnotami, popismi, povinnými poľami.
  • Cache drahých fetchov s rozumným TTL (u nás 5 min).
  • Discovery tool pre získanie schemy (get_component_info) — lacnejšie ako skúšať parametre.
  • Štruktúrované chyby s návrhom riešenia — náš get_icon pri chybe vráti najbližšie alternatívy.
  • Curated subset, nie všetko. Vystavujeme len ~186 ikoniek, nie celé FA Pro.

 Nefunguje

  • Stovky toolov v jednom serveri — LLM sa stratí a začne náhodne tipovať.
  • Vágne názvy ako do_thing, process_data, handle_request.
  • Jeden mega-tool s 30 parametrami a režimami. Lepšie ich rozdeliť do menších.
  • Obrovské nestrukturované JSON blob odpovede — zaberú kontext bez prínosu.
  • Popis bez príkladu — LLM zvládne aj abstraktný popis, ale s príkladom je úspešnosť výrazne vyššia.
  • Tichý fail — návrat null alebo prázdneho stringu bez vysvetlenia.
§ 05 · Kedy použiť

Kedy MCP, kedy nie.

Nie každý kontextový problém potrebuje vlastný MCP server. Tu sú konkrétne príklady — kedy sa to oplatí a kedy len pridáme latency a komplexitu.

 Vhodné použitia

  • Interný design system — komponenty, tokens, ikonky. Veci, ktoré LLM nemá v tréningových dátach.
  • Interné dáta a API — customer-by-ID lookup, support history, štruktúra firmy. Citlivé veci čo sa do tréningových dát nedostanú.
  • External SaaS s API kľúčom — Sentry issues, Linear tickety, Grafana queries, Slack messages. LLM ich vidí cez náš server, nie cez kopírovanie do promptu.
  • Codebase navigation — symbol search, file dependencies, recent commits. Repo-specific kontext.

 Nevhodné použitia

  • Verejné API LLM pozná — Stripe, AWS, npm packages, MDN. MCP wrapper by bol len indirection cez známe.
  • Čistá výpočtová logika — regex, parsing, formátovanie. LLM to spočíta priamo, MCP len pridá latency.
  • Heavy data dumps — celé DB tabuľky, gigabajty logov. Zaberie kontext bez prínosu — lepšie urobiť tool ktorý vráti agregát alebo filter.
  • Real-time produkčné mutácie — DB writes, deploy commandy, finančné transakcie. Bez human-in-the-loop priveľa rizika.
Heuristika. Ak by sa odpoveď dala vygúglovať a LLM by ju vedel reprodukovať z hlavy, MCP nie je riešenie. Ak je za firewallom alebo sa mení každú hodinu, MCP áno.
§ 06 · Bezpečnosť

Štyri vrstvy obrany.

MCP server nie je len helper — môže byť útočným vektorom. Treba myslieť na bezpečnosť od transportu cez auth, validáciu inputov, až po operations.

01 · Transport

Šifrované spojenie

stdio je process-isolated — lokálny Docker = bezpečný default. HTTP mode potrebuje TLS: Let's Encrypt certifikát + reverse proxy (čo aj robíme pri vzdialenom deploy).

02 · Auth

Autentifikácia klientov

stdio: process boundary stačí — kto má docker exec, má prístup. HTTP: API kľúč v hlavičke, OAuth scope, alebo capability tokens pre granular access.

03 · Tool design

Validácia inputov

Striktné JSON schema s enum hodnotami a type checks. Žiadny path traversal pri filesystem readoch. Žiadne SQL ani shell injection cez user-supplied strings.

04 · Operations

Audit + human-in-the-loop

Logy čo / kedy / kým bolo zavolané. Mutácie produkcie iba s explicitnou konfirmáciou používateľa. Rate limit pre runaway agents (loop útok).

Pozor. Aj samotné popisy toolov sú attack surface — LLM ich číta a riadi sa nimi. Prompt injection cez tool description je reálny vektor. Popisy by mali pochádzať len z dôveryhodných zdrojov, nie z user-generated content.
§ 07 · Motivácia

Prečo sme to postavili?

Náš styleguide nie je v Claudových tréningových dátach. Bez MCP teda Claude pri otázke „daj mi tlačidlo" generuje presvedčivo vyzerajúce HTML — také, aké videl na tisícoch iných webov.

Stav A · bez MCP

Hallucinated kód

  • — Claude navrhuje komponenty zo svojej hlavy
  • — Vymyslené CSS triedy, nesprávna štruktúra
  • — Vývojár musí ručne opravovať každý fragment
  • — Každý má svoju verziu „toho istého" tlačidla
  • — Žiadny vzťah ku skutočnému styleguide
Stav B · s MCP

Kanonický kód

  • ✓ Claude sa opýta nášho servera
  • ✓ Vráti skutočný HTML zo styleguide
  • ✓ Správne triedy, správna štruktúra — hneď
  • ✓ Jeden zdroj pravdy pre AI aj ľudí
  • ✓ Update styleguide = update pre AI naraz
§ 08 · Naša implementácia

Osem toolov, osem otázok.

Každý tool rieši jednu konkrétnu otázku Claudea o našom dizajn systéme.

get_setup

HTML tagy pre načítanie všetkých styleguide assetov.

language

get_starter_template

Kompletný HTML dokument pripravený na použitie.

language title

get_component

Renderuje HTML fragment komponentu s parametrami.

name ...

list_components

Vypíše všetky dostupné komponenty s popisom.

type

get_component_info

Detail komponentu — schéma parametrov.

name

list_icons

~186 Font Awesome ikoniek v 5 štýloch.

style

get_icon

HTML snippet pre ikonku, s návrhom alternatív.

name style

get_utilities

Utility triedy: flex, spacing, colors, sizing.

category
Internal only. V lokálnom Docker mode sú navyše dostupné tooly, ktoré čítajú priamo zo zdrojov repozitára: list_scss_utilities, find_scss_class, get_data_structure.
§ 09 · Architektúra

Jeden server, dva módy.

Rovnaký codebase, dve prostredia: externé fetch z produkcie cez HTTP, alebo lokálne čítanie zo súborov.

External mode

default · beží kdekoľvek

Server nemá prístup k repozitáru. Pri každom requeste si stiahne aktuálne dáta z mrtns.sk/assets a 5 minút si ich drží v cache. Vždy konzistentné s produkciou.

Claude → MCP server → mrtns.sk
Internal mode

dev · lokálny Docker

Aktivuje sa nastavením MARTINUS_STYLEGUIDE_APP_DIR. Server číta priamo zo súborov repozitára — zmena v app/mcp-components.yaml je vidieť okamžite, bez deploya.

Claude → MCP server → app/*.yaml
