§ 01 · Motivácia

Prečo sme to postavili?

Náš styleguide nie je v Claudových tréningových dátach. Bez MCP teda Claude pri otázke „daj mi tlačidlo" generuje presvedčivo vyzerajúce HTML — také, aké videl na tisícoch iných webov. V príkladoch používame Claude, ale to isté platí pre každý AI nástroj, ktorý hovorí MCP.

Stav A · bez MCP

Vymyslený kód (halucinácie)

  • — Claude navrhuje komponenty zo svojej hlavy
  • — Vymyslené CSS triedy, nesprávna štruktúra
  • — Vývojár musí každý kúsok kódu ručne opraviť
  • — Každý má svoju verziu „toho istého" tlačidla
  • — Žiadny vzťah ku skutočnému styleguide
Stav B · s MCP

Kanonický kód

  • ✓ Claude sa opýta nášho servera
  • ✓ Vráti skutočný HTML zo styleguide
  • ✓ Správne triedy, správna štruktúra — hneď
  • ✓ Jeden zdroj pravdy pre AI aj ľudí
  • ✓ Update styleguide = update pre AI naraz
§ 02 · Koncept

Čo je Model Context Protocol?

Veľké jazykové modely vedia rozprávať, ale samé osebe sú v izolácii. Nevidia naše súbory, databázy, ani styleguide. MCP je protokol, ktorý túto bariéru ruší — definuje, ako môže AI volať externé nástroje a získavať dáta z reálneho prostredia.

Pred MCP každý vendor (OpenAI, Google, Anthropic) riešil prepojenie modelu s nástrojmi po svojom — proprietárne function calling API. Výsledok? Roztrieštený ekosystém, integrácie sa nedali zdieľať medzi modelmi, každá appka si musela písať vlastné lepidlo.

V novembri 2024 Anthropic publikoval MCP ako open-source, vendor-neutrálny štandard. Princíp je rovnaký ako pri Language Server Protocole (LSP) pre IDE — jeden protokol, ktorý hovoria všetci.

Ako USB-C. Zariadenia sa po zapojení najprv „pokecajú“ — čo vieš ty, čo viem ja: nabíjanie, dáta, obraz — a podľa toho nastavia spoluprácu. MCP robí to isté: server pri pripojení povie, aké nástroje ponúka a ako sa s nimi pracuje.

§ 03 · Stavebné bloky

MCP server ponúka tri veci.

Tools, resources, prompts — tri stavebné bloky každého servera. My zatiaľ používame len tools.

U nás 01 · Aktívne

Tools

Funkcie, ktoré AI volá s parametrami. Server ich vykoná a vráti výsledok. Najpoužívanejší z trojice.

get_component(name: "buttons")

Pasívne 02 · Dáta

Resources

Dáta, ktoré server vystavuje na čítanie — súbory, log záznamy. AI ich číta podľa URI.

file:///app/main.scss

Šablóny 03 · Prompty

Prompts

Pripravené prompt šablóny s parametrami, ktoré klient ponúka ako rýchle akcie.

"Vytvor formulár pre {{entita}}"

Transport. MCP definuje dva transporty: lokálny stdio (server beží ako proces, komunikuje cez pipy) a vzdialený HTTP so server-sent events. Aplikačná vrstva je JSON-RPC 2.0.
§ 04 · Lifecycle

Anatómia jedného volania.

Čo sa stane medzi promptom a odpoveďou. Discovery (1–2) sa deje raz pri pripojení; tool call (3–5) môže prebehnúť mnohokrát.

1 initialize

Úvodný „pokec“ — klient a server si povedia, akú verziu a aké funkcie podporujú. Ako USB-C pri zapojení.

2 tools/list

Klient si vypýta zoznam dostupných toolov so schemami parametrov.

3 AI rozhoduje

AI na základe promptu vyberá, ktorý tool zavolá a s akými argumentmi.

4 tools/call

Klient pošle request s názvom toolu a parametrami. Server validuje a vykoná.

5 response

Server vráti štruktúrovaný výsledok, ktorý AI použije v odpovedi.

Tip. Celý cyklus prebehne v jednej konverzácii niekoľkokrát. Server beží stále — žiadny cold start.

§ 05 · Praktiky

Čo funguje, čo nie.

MCP je jednoduchý protokol, ale postaviť dobrý server je iný príbeh. AI nečíta kód — číta popisy toolov.

 Funguje

  • Stručné, popisné názvy — get_component, list_icons. AI ich číta ako API.
  • Striktné JSON schema pre parametre s enum hodnotami, popismi, povinnými poľami.
  • Cache — pomalé sieťové dáta si server pamätá (u nás 5 minút).
  • Discovery tool pre získanie schemy (get_component_info) — lacnejšie ako skúšať parametre.
  • Štruktúrované chyby s návrhom riešenia — náš get_icon pri chybe vráti najbližšie alternatívy.
  • Curated subset, nie všetko. Vystavujeme len ~186 ikoniek, nie celé FA Pro.

 Nefunguje

  • Stovky toolov v jednom serveri — AI sa stratí a začne náhodne tipovať.
  • Vágne názvy ako do_thing, process_data, handle_request.
  • Jeden mega-tool s 30 parametrami a režimami. Lepšie ich rozdeliť do menších.
  • Obrovské neštruktúrované odpovede — zaberú kontext bez prínosu.
  • Popis bez príkladu — AI zvládne aj abstraktný popis, ale s príkladom je úspešnosť výrazne vyššia.
  • Tichý fail — návrat null alebo prázdneho stringu bez vysvetlenia.
§ 06 · Kedy použiť

Kedy MCP, kedy nie.

Nie každý kontextový problém potrebuje vlastný MCP server. Tu sú konkrétne príklady — kedy sa to oplatí a kedy len pridáme latency a komplexitu.

 Vhodné použitia

  • Interný design system — komponenty, tokens, ikonky. Veci, ktoré AI nemá v tréningových dátach.
  • Interné dáta a API — vyhľadanie zákazníka, história supportu, štruktúra firmy. Citlivé veci, ktoré sa do tréningových dát nedostanú.
  • Externé služby s API kľúčom — Sentry, Linear, Grafana, Slack. AI ich vidí cez server, netreba nič kopírovať do promptu.
  • Orientácia v kóde — kde je čo definované, čo na čom závisí, posledné commity.

 Nevhodné použitia

  • Verejné API, ktoré AI pozná — Stripe, AWS, npm, MDN. MCP by bol len zbytočná obchádzka.
  • Čistá výpočtová logika — regex, parsing, formátovanie. AI to spočíta priamo, MCP len pridá zdržanie.
  • Obrovské dáta — celé DB tabuľky, gigabajty logov. Zaberú kontext bez prínosu — lepší je tool, ktorý vráti súhrn alebo filter.
  • Zápisy do produkcie — zmeny v DB, deploye, finančné transakcie. Bez potvrdenia človekom priveľké riziko.
Heuristika. Ak by sa odpoveď dala vygúglovať a AI by ju vedela reprodukovať z hlavy, MCP nie je riešenie. Ak je za firewallom alebo sa mení každú hodinu, MCP áno. MCP s API pritom nesúperí — tool ich zvyčajne len obaľuje pre AI.
§ 07 · Bezpečnosť

Štyri vrstvy obrany.

MCP server nie je len pomocník — môže byť aj vstupnou bránou pre útok. Treba myslieť na bezpečnosť od spojenia cez prihlasovanie a validáciu vstupov až po prevádzku.

01 · Transport

Šifrované spojenie

Lokálny server (stdio) beží izolovane v Dockeri — bezpečný default. Vzdialený (HTTP) potrebuje TLS: Let's Encrypt certifikát + reverse proxy (čo robíme pri vzdialenom deployi).

02 · Auth

Autentifikácia klientov

Lokálne: kto má prístup ku kontajneru (docker exec), má prístup k serveru. Vzdialene: API kľúč v hlavičke alebo OAuth.

03 · Tool design

Validácia inputov

Striktná schéma s vymenovanými hodnotami a kontrolou typov. Text od používateľa nikdy nejde priamo do ciest k súborom, SQL ani shellu.

04 · Operations

Audit + potvrdenie človekom

Logy čo / kedy / kto zavolal. Zmeny v produkcii iba s explicitným potvrdením používateľa. Limit počtu volaní, keby sa agent zacyklil.

Pozor. Aj samotné popisy toolov sú útočná plocha — AI ich číta a riadi sa nimi. Škodlivá inštrukcia ukrytá v popise (prompt injection) je reálny útok. Inštalujme len servery z dôveryhodných zdrojov.
§ 08 · Naša implementácia

Osem toolov, osem otázok.

Každý tool rieši jednu konkrétnu otázku AI o našom dizajn systéme.

get_setup

HTML tagy pre načítanie všetkých styleguide assetov.

language

get_starter_template

Kompletný HTML dokument pripravený na použitie.

language title

get_component

Renderuje HTML fragment komponentu s parametrami.

name ...

list_components

Vypíše všetky dostupné komponenty s popisom.

type

get_component_info

Detail komponentu — schéma parametrov.

name

list_icons

~186 Font Awesome ikoniek v 5 štýloch.

style

get_icon

HTML snippet pre ikonku, s návrhom alternatív.

name style

get_utilities

Utility triedy: flex, spacing, colors, sizing.

category
Internal only. V lokálnom Docker mode sú navyše dostupné tooly, ktoré čítajú priamo zo zdrojov repozitára: list_scss_utilities, find_scss_class, get_data_structure.
§ 09 · Architektúra

Jeden server, dva módy.

Rovnaký codebase, dve prostredia: externé fetch z produkcie cez HTTP, alebo lokálne čítanie zo súborov.

External mode

default · beží kdekoľvek

Server nemá prístup k repozitáru. Pri každom requeste si stiahne aktuálne dáta z mrtns.sk/assets a 5 minút si ich drží v cache. Vždy konzistentné s produkciou.

AI klient → MCP server → mrtns.sk
Internal mode

dev · lokálny Docker

Aktivuje sa nastavením MARTINUS_STYLEGUIDE_APP_DIR. Server číta priamo zo súborov repozitára — zmena v app/mcp-components.yaml je vidieť okamžite, bez deploya.

AI klient → MCP server → app/*.yaml
